ThreadSafeQueueLib: Lock-Free MPMC Queue in C++

That time I got reincarnated as a lock-free queue library implementer. This project turned into a serious lesson in memory ordering, false sharing, and how much operating systems background helps once concurrency stops being theoretical.

Table of Contents

Overview

I started this project to understand concurrent queue design properly instead of repeating stock phrases about lock-free code being fast. Very quickly, it became clear that the interesting part is not just the queue itself. It is the combination of interface design, cache behavior, progress guarantees, and the memory model.

Motivation

A standard queue interface becomes fragile under contention. Something as ordinary as checking empty() before front() and pop() is not a real plan once another thread can mutate the structure in between. I wanted to understand what a safer interface and a faster implementation actually demand.

Design Decisions

Why lock-free?

Mutex-based queues are much easier to write and reason about. The lock-free route only makes sense if the learning goal is precisely to understand progress guarantees, contention, and the parts of the design where performance meets correctness.

Bounded versus unbounded

Bounded queues are attractive because the memory is fixed and predictable. Unbounded queues avoid the full-buffer problem but push you toward allocation and reclamation concerns that are easy to underestimate.

Compile-time configuration

I prefer selecting queue behavior through templates rather than runtime polymorphism. The compiler should know what variant is being built and optimize around those assumptions directly.

The Hard Parts

Memory ordering

The C++ memory model is where this project stopped being theoretical. Using seq_cst everywhere is not design, it is temporary avoidance. The real work is deciding where acquire-release is enough and where relaxed operations are still correct.

False sharing

Producer and consumer indices sharing a cache line can quietly destroy throughput through invalidation traffic. Once you see it in practice, "aligning state properly" stops sounding like a small optimization and starts looking like basic queue hygiene.

Debugging concurrency

Concurrency bugs are good at creating false confidence. A thousand clean runs do not mean much. Stress tests, invariant checks, and ThreadSanitizer matter because the failure cases are exactly the ones you do not see often enough.

Behind the Project

The project also pushed me back into systems reading. I needed stronger OS and architecture intuition to reason cleanly about what the queue was doing under the hood. That is why this post naturally links to the Computer Architecture & OS note and the broader C++ series.

What Is Next

I still want deeper evaluation, more benchmarking, and cleaner write-ups around the parts that currently live as implementation notes. The goal is not just a queue that works, but a queue that I can explain without hand-waving the hard bits away.
