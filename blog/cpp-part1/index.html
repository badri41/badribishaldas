C++ Part 1: The Beginning of the End

That time I got reincarnated as a C++ dev. This is the first note in a series about the resources I trust, the books I keep nearby, and the parts of the language that still feel worth wrestling with.

Table of Contents

Overview

I wanted one place on the site where every C++ part could live without being buried inside a generic list of posts. This is that starting point. The point is not to pretend I already understand the whole language. The point is to track the resources, questions, and implementation details that are actually moving my understanding forward.

Where I Am Learning From

I keep coming back to resources that are practical, opinionated, and written by people who clearly use C++ for real work instead of treating it like a museum piece.

  • C++ Weekly for tight modern C++ details.
  • CppCon for both deep talks and the back-to-basics sessions.
  • The Cherno when I want a faster explanation before going deeper elsewhere.
  • Mike Shah for systems-heavy material.
  • NotesByNick for focused topic breakdowns.

What helps the most

The best resources are the ones that connect language rules back to actual program behavior: performance, object lifetime, undefined behavior, API design, and tradeoffs that survive outside toy examples.

Helpful Books

  • Effective Modern C++ by Scott Meyers.
  • Beautiful C++.
  • C++20 Software Design by Klaus Iglberger.
  • C++ Templates: The Complete Guide, which feels more like a long-term project than a casual read.
  • C++ Concurrency in Action by Anthony Williams.

Current State

Right now, a lot of my C++ learning is tied directly to ThreadSafeQueueLib. That project pushed me toward the C++ memory model, lock-free data structures, and eventually toward operating systems and architecture material because the concurrency side gets much clearer when the systems background is stronger.

That is why the systems reading list exists next to this series. The two are linked. The project raises questions, the systems background answers some of them, and then C++ raises newer and worse questions.

What Comes Next

The next parts in this series should get narrower and more useful. The most natural follow-ups are:

  • Virtual destructors and object lifetime.
  • Public inheritance versus private inheritance.
  • The memory model, especially where intuition breaks down.
  • Templates and API design for code that wants zero-cost abstractions without becoming unreadable.
