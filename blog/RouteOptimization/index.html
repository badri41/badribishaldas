RouteOptimizationChallenge: A HVRPSTDW Solver in C++

From Jan 2026 to Mar 2026, at Kriti 26' (IIT Guwahati), our hostel team of 6 (me as project head, 2 sophomores, 3 freshers) built a full-stack web application for a hard optimization track focused on employee transportation planning. The core was a C++ solver for the Heterogeneous Vehicle Routing Problem with Simultaneous Pickup and Delivery, Time Windows, and Driver Constraints (HVRPSTDW).

Overview

This challenge combined product engineering and algorithmic optimization. We had to deliver a usable web interface while also designing a competitive solver for an NP-hard routing problem. The task was not just "find routes", but find cost-efficient, feasible routes across a heterogeneous fleet under realistic operational constraints. The codebase for this work is available in the VeloraMobilityOptimizer repository, and the frontend is available at the live Render deployment.

Problem Setting

The optimization target was corporate employee transport. Vehicles had different capacities and costs, requests included simultaneous pickup and delivery behavior, and schedules needed to respect soft time windows and driver-side feasibility constraints. We modeled violations through penalties, turning route quality into a penalty-aware objective instead of a simplistic distance-only target.

Solver Design

Hybrid Metaheuristic in C++

We designed a hybrid metaheuristic solver in C++ to balance solution quality and runtime. Exact methods were not practical at competition scale, so the focus was on generating high-quality feasible plans quickly and then iteratively improving them.

Phase 1: Initial Solution Construction

We used a multi-start Solomon I1-style greedy insertion approach with up to 20 restarts. Each restart shuffles requests inside priority buckets, then inserts them one-by-one at the cheapest feasible position across all vehicles. The best solution among all restarts is retained as the starting point for downstream improvement.

Phase 2: Simulated Annealing Improvement

We then ran deterministic Simulated Annealing (seed = 42) with Metropolis acceptance and a maxNoImprove stopping condition, so runs remain reproducible. Temperature scales with instance complexity: T0 = min(0.3 x cost, 1000), cooling alpha = 0.99, with adaptive reheating every maxNoImprove / 3 stagnant steps.

Neighborhood Operators

The SA phase uses 5 weighted operators: Greedy Relocate (28%), Intra-Route Relocate (28%), Exchange (18%), 2-Opt (13%), and Or-Opt (13%). This gives a stable balance between route-level restructuring and fine local edits.

Phase 3: Adaptive Large Neighborhood Search (ALNS)

After SA, we run a time-bounded ALNS destroy-repair loop using Ropke and Pisinger style adaptive scoring with sigma1 = 33, sigma2 = 9, sigma3 = 3, and reaction factor lambda = 0.8. Destroy operators are: Shaw Removal (seed + nearest related requests), Random Removal (diversification), and Route Removal (drop the most-penalized route to escape plateaus). Repair is performed with Regret-2 insertion, prioritizing requests with the largest gap between second-best and best insertion cost. In practice this pushes tightly constrained requests earlier and improves robustness under soft-window penalties.

Phase 4: Post-ALNS SA Polish

We restart SA from the ALNS best solution with a shorter maxNoImprove budget for fine-grained polishing. This phase focuses on last-mile local gains, then keeps whichever solution is better between ALNS output and post-ALNS SA.

Objective and Constraints

The objective combined routing cost and penalty terms for soft-constraint violations. The solver continuously traded off feasibility pressure and cost pressure, which was essential for converging toward practically valid schedules under tight constraints.

Distance Computation

Distances are precomputed once into an N x N matrix and then read in O(1) during SA and ALNS. For realistic travel-time geometry, we use OSRM Table API in a single matrix request (roughly 5-20s precompute depending on instance size). For quick iteration, we also support Haversine straight-line distances.

Results

Across benchmark cases, the final solver achieved roughly 74.9%-88.6% lower transport cost than baseline. We also observed fleet usage reductions of up to 50% on selected instances. The results validated the full 4-phase pipeline: multi-start construction, SA exploration, ALNS diversification, and post-ALNS local polish.

Technical Report

We documented the full formulation and methodology in a 25-page technical report, including constraints, objective design, heuristic details, and experimental behavior. Writing the report was a major part of the project because it forced us to justify each modeling and optimization choice rigorously.

What I Learned

This challenge taught me that good optimization systems are as much about modeling discipline as algorithm selection. A clean objective, realistic penalties, and robust neighborhood design mattered more than chasing complicated machinery. It also reinforced how useful C++ is for fast iterative solvers when you need performance and control.
