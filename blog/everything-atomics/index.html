Before Atomics: Why Reordering Exists

CPUs and compilers are built for throughput, not source-level instruction order. They exploit independent work, hide memory latency, and keep pipelines busy. The key rule is if the single-thread behaviour is preserved, they can reorder instructions freely.

Compiler Reordering

The compiler can reorder independent instructions for optimization, register allocation, and scheduling. Without atomic operations or fences, it does not need to preserve inter-thread intent.

CPU Out-of-Order Execution

Hardware may execute independent instructions early and retire them in a way that preserves local program semantics. Internally, execution order can differ from source order.

Memory Reordering (The Concurrency Problem)

Across cores, loads and stores can become visible in unexpected orders due to store buffers, cache effects, and architecture-specific rules. This is the reordering that breaks naive shared-state code.

Store Buffering

where each core keeps recent writes in a private buffer until they are flushed to the shared cache

Cache Coherence and MESI (Quick Intuition)

Cache coherence is the mechanism that keeps copies of the same memory location consistent across cores. If Core 1 updates a cache line, other cores must not keep reading stale values forever. Coherence protocols coordinate this process using line states and invalidations.

A simple mental model is MESI: Modified, Exclusive, Shared, Invalid. A write often forces invalidations on other cores, and that coherence traffic can become expensive under contention. This is one reason lock-free code can still perform badly when many threads constantly fight over the same cache line.

False Sharing and C++17 Interference Sizes

False sharing happens when two independent variables used by different threads are placed on the same cache line. Even though the variables are logically unrelated, writes from one thread invalidate the line for the other thread, causing heavy coherence traffic and performance collapse.

C++17 provides std::hardware_destructive_interference_size in <new>. It represents a conservative spacing value (commonly 64-256 bytes depending on platform) to separate hot fields and reduce false sharing. 

#include <atomic>
#include <new>

struct alignas(std::hardware_destructive_interference_size) Counter {
    std::atomic<std::uint64_t> value{0};
};

Counter producer_counter;
Counter consumer_counter;

The idea is simple: keep frequently written per-thread data far enough apart so they do not ping-pong the same cache line.

The Happens-Before Relation in C++

In concurrent code, assume operations can be observed out of order unless you create ordering explicitly. In C++, that ordering is expressed through the happens-before relation.

If A happens-before B, then:

  • A's effects are visible to B.
  • B cannot observe a state before A.

If conflicting writes and accesses are not ordered by happens-before, that is a data race and the program has undefined behavior.

C++ Memory Model in One View

Since C++11, the language defines atomic operations and ordering guarantees independent of any one compiler or CPU. Atomics in <atomic> provide race-free shared access when used correctly.

But what is Memory Ordering?

Memory ordering is about controlling visibility and ordering of operations across threads. It defines how one thread's writes become visible to another thread's reads. Sure the atomic variable is lock-free, but suppose the atomic variable controls a non-atomic data/memory. Read/write to the non-atomic data can be reordered and there is no guarantee that the other thread will see the updated data when it sees the updated atomic variable. This is where memory ordering comes into play.

The Memory Orders

  • memory_order_relaxed: atomicity only, no inter-thread ordering.
  • memory_order_consume: data-dependency ordering; effectively treated as acquire by mainstream compilers today.
  • memory_order_acquire: prevents later operations from moving before the load/RMW.
  • memory_order_release: prevents earlier operations from moving after the store/RMW.
  • memory_order_acq_rel: both acquire and release on RMW operations.
  • memory_order_seq_cst: strongest model, including a single global order of seq_cst operations.

Acquire-Release: The Core Pair

Release on the producer and acquire on the consumer is the standard publish-subscribe pattern. If an acquire load sees the value written by a release store, then all writes sequenced-before that release become visible. 

std::atomic<bool> ready{false};
int data = 0;

// Producer
void t1() {
    data = 42;
    ready.store(true, std::memory_order_release);
}

// Consumer
void t2() {
    while (!ready.load(std::memory_order_acquire)) {}
    // guaranteed to see data == 42
    std::cout << data << '\n';
}

What seq_cst Adds

seq_cst gives acquire/release style barriers plus one extra global property: all seq_cst operations appear in a single total order seen consistently by all threads. This simplifies reasoning and rules out some outcomes allowed by weaker orders.

Acquire-Release vs Sequential Consistency

Acquire-Release

Guarantees:

  • Creates happens-before between specific synchronizing threads.
  • Ensures visibility of prior writes across that synchronization edge.

But:

  • No single global ordering across all threads.
  • Different threads may observe operations in different relative orders.

Sequential Consistency (seq_cst)

Guarantees:

  • All seq_cst operations appear in one global total order that every thread agrees on.
  • Still retains acquire/release-style ordering for those operations.

In short: seq_cst adds a global ordering constraint on top of acquire-release behavior.

When acq_rel Is Enough

Use acq_rel for read-modify-write operations when you need two-way ordering, but do not need the global total order guarantee of seq_cst. Common examples include fetch_add and compare-exchange loops in lock-free algorithms.

Relaxed Ordering: Useful but Easy to Misuse

Relaxed operations are atomic but do not publish ordering to other memory locations. Great for counters, IDs, and statistics where relative order does not matter. Wrong choice for "flag means data is ready" signaling.

CAS (Compare-And-Swap) Properly Understood

Compare-and-swap updates atomics conditionally. 

bool ok = x.compare_exchange_strong(expected, desired);
  • If x == expected, writes desired and returns true.
  • If not equal, writes current x into expected and returns false.

The Important CAS Detail

On failure, expected is overwritten with the observed value. This is why retry loops work naturally: 

int expected = old_value;
while (!x.compare_exchange_weak(expected, new_value,
                                std::memory_order_acq_rel,
                                std::memory_order_acquire)) {
    // expected is refreshed for the next attempt
}

Weak vs Strong: Retry vs Decision Semantics

compare_exchange_weak is allowed to fail spuriously, meaning it may report failure even when the current value equals the expected value. In contrast, compare_exchange_strong fails only when the comparison genuinely fails.

This leads to two usage patterns. compare_exchange_weak is designed for retry semantics and is typically used in a loop where failure (real or spurious) simply triggers another attempt. 

int expected = old_value;
while (!x.compare_exchange_weak(expected, new_value,
                                std::memory_order_acq_rel,
                                std::memory_order_acquire)) {
    // retry
}

In this pattern, spurious failure is harmless. More importantly, when failure is due to a real mismatch, it means another thread has modified the shared state. That is the core lock-free progress idea: one thread may fail, but some thread is making progress.

compare_exchange_strong is better for decision semantics, where one attempt decides control flow. 

if (x.compare_exchange_strong(expected, new_value)) {
    // success path
} else {
    // definite mismatch
}

A failure here reliably means the expected value did not match, not an artifact of spurious failure.

Important caveat: retry-based designs can still spin indefinitely if the expected value never becomes true again. ), but it shows that lock-free does not imply fairness or starvation-freedom for every individual thread.

Progress Guarantees: Lock-Free vs Wait-Free

  • Lock-free: system-wide progress, at least one thread completes in finite steps.
  • Wait-free: per-thread progress, every thread completes in finite steps.

Wait-free is stronger. It is about bounded steps, not wall-clock time.
