C++ Episode 2: Runtime Dispatch & Polymorphism

Episode 2 is about one core systems question: how do we handle behavior through a common interface when the implementation is only known at runtime? This post covers virtual dispatch, vtables, RTTI, type erasure, and when alternatives like CRTP or std::visit are a better fit.

Table of Contents

Why Runtime Dispatch Exists

We often have related types and want one way to process all of them: 

std::vector<Vehicle*> types;
for (auto* type : types) {
    type->accelerate();
}

At compile time, each element has static type Vehicle*. At runtime, the pointed object may be Car, Bike, or something else. Runtime dispatch decides which concrete implementation should execute.

C++'s Answer: Inheritance + Dynamic Binding

In classic C++, runtime dispatch is built from inheritance and virtual functions: 

class Vehicle {
public:
    virtual void accelerate() = 0;
    virtual ~Vehicle() = default;
};

class Car : public Vehicle {
public:
    void accelerate() override {
        // car-specific acceleration
    }
};

A call through Vehicle* is resolved at runtime (runtime binding). The mechanism underneath is usually a per-type vtable and a hidden vptr inside polymorphic objects.

What Is a Vtable, Really?

A useful mental model: a vtable is just a structure of function pointers. 

struct vtable {
    void (*accelerate)(void*);
};

template <typename T>
vtable vtable_for = {
    [](void* obj) {
        static_cast<T*>(obj)->accelerate();
    }
};

Conceptually, a call site does three steps: read vptr, fetch function pointer from vtable slot, make an indirect call. That flexibility is powerful, but it has overhead.

What the Compiler Typically Emits

On many targets, a virtual call roughly looks like this at assembly level: 

mov rax, [rdi]        // load vptr from object
call [rax + offset]   // indirect call via vtable slot

Exact instructions vary by compiler, optimization level, and ABI, but the indirect-call shape is the key idea.

Cost Model: Hot Path vs Cold Path

  • Extra pointer in each polymorphic object (the vptr).
  • Indirect call through function pointer (harder for optimizer/inlining).
  • Potential branch prediction and cache effects on hot paths.

Critical distinction: in hot loops, virtual dispatch can matter; in cold paths, it is often irrelevant compared to I/O or higher-level latency. Measure before optimizing.

Devirtualization: When Virtual Becomes Direct

Compilers can sometimes remove virtual overhead by proving the dynamic type at compile time or at link time. This optimization is called devirtualization.

  • Common enablers: final, whole-program optimization/LTO, and visible concrete types.
  • Result: a direct call (sometimes fully inlined) instead of vtable lookup.

Virtual Destructor and override Matter

If a base class is intended for polymorphic deletion, it must have a public virtual destructor. Otherwise deleting a derived object through a base pointer is undefined behavior. A virtual destructor ensures the correct derived destructor runs. 

class Base {
public:
    virtual ~Base() = default;
    virtual void f();
};

class Derived : public Base {
public:
    void f() override; // compile-time check for intended override
};

override is optional but strongly recommended because it prevents silent mismatches in function signatures.

RTTI, Upcasting, and Downcasting

RTTI (Run-Time Type Information) lets you inspect dynamic type at runtime in polymorphic hierarchies. Upcasting (Derived* to Base*) is safe. Downcasting needs runtime checks. 

Base* b = new Derived();

// Safe checked downcast
if (Derived* d = dynamic_cast<Derived*>(b)) {
    d->f();
}

If RTTI is disabled (for example with -fno-rtti), tools like dynamic_cast and typeid become unavailable for this purpose.

ABI and Binary Compatibility (Why This Matters in Real Systems)

Vtable la yout is ABI-dependent, meaning details differ across compilers/platform ABIs. Still, virtual interfaces are heavily used for binary boundaries in:

  • plugin systems (host loads implementations at runtime),
  • shared libraries (behavior exposed behind stable interfaces).

This is where MLI becomes practical: runtime polymorphism helps independently built modules interoperate cleanly.

Type Erasure: Polymorphism Without Inheritance

Type erasure hides concrete types behind a stable non-template interface, while forwarding behavior internally. The concrete type is erased from the public API, not from runtime behavior.

Standard examples include std::function, std::any, and pointer-erased ownership forms. A common implementation pattern is:

  1. Public wrapper object (non-template API).
  2. Internal abstract concept interface.
  3. Templated model storing the concrete object and forwarding calls.

Small Buffer Optimization (SBO)

To avoid heap allocations for small objects, many erased wrappers keep an inline buffer and only allocate when object size exceeds that buffer. 

union {
    void* ptr_;
    std::aligned_storage_t<16> buffer_;
};

// Small object case
on_heap_ = false;
new (&buffer_) Any{vehicle}; // placement new

Libraries like std::function commonly rely on SBO for small lambdas to improve locality and avoid allocation overhead.

Alternatives to Virtual Dispatch

  • CRTP (Curiously Recurring Template Pattern) for compile-time polymorphism.
  • std::variant + std::visit for closed sets of types.
  • constexpr and templates for compile-time computation paths.

Will explain in detail about CRTP in a future post, but the key idea is to use templates to achieve polymorphic behavior without virtual functions, at the cost of compile-time flexibility.

The Dispatch Landscape (Big Picture)

C++ offers multiple dispatch models, each with different trade-offs in flexibility, performance, and design.

DRY (Don’t Repeat Yourself): avoid duplicating type-specific logic.
RTB (Runtime Binding): call target resolved at runtime (virtual/vtable path).
OFD (Open for Extension, Closed for Modification): add behavior without rewriting existing paths.
MLI (Multiple Library Integration): modules and libraries interact through stable interfaces.
AED (Adding Extra Data): design can evolve with new behavior/data over time.

Compiler Optimization Flags in Context

Optimization level affects how much overhead survives in generated code:

  • -O0: almost no optimization, best for debugging.
  • -O1: basic optimization.
  • -O2: stronger general optimizations.
  • -O3: aggressive optimizations, often including more inlining/vectorization.
  • -Os: optimize for size.

Design Principle Anchor

  • LSP (Liskov Substitution Principle): If S is a subtype of T, then objects of type T should be replaceable with objects of type S without breaking the program.

Closing Notes

Runtime dispatch is a practical engineering tool, not just a textbook OOP concept. If you understand its mechanics (vtable, RTTI, erased wrappers, SBO, ABI tradeoffs, and devirtualization), you can choose it intentionally and avoid both premature abstraction and premature optimization.
